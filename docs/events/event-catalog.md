# Event Catalog

## Purpose

This document defines the event types produced by the Microsoft Fabric Smart Farming Analytics Platform.

The Event Catalog serves as the authoritative reference for all streaming events generated by the platform. It describes the purpose of each event, the business entity producing the event, the producing system, the primary consumers, and the expected event frequency.

All events conform to the Canonical Event Contract defined for this project.

---

# Canonical Event Contract

Every event generated by the platform shall include the following metadata fields.

| Field               | Description                                                   |
| ------------------- | ------------------------------------------------------------- |
| event_id            | Globally unique identifier for the event                      |
| event_type          | Identifies the business event (`environmental`, `equipment`, `CropLifecycleEvent`, `CropTelemetryEvent`, `IrrigationTelemetryEvent`, `LightingTelemetryEvent`, `MaintenanceEvent`, `FacilityEvent`) |
| timestamp           | UTC ISO timestamp when the event occurred                     |
| schema_version      | Version of the event schema (`1.0`)                           |
| facility_id         | Unique identifier for the farming facility (`FAC-001` through `FAC-008`) |
| zone_id             | Climate-controlled growing zone (`ZONE-000` through `ZONE-010`) |
| operator_contact    | Technician contact email (`tech.fac-001@smartfarm.ph`) for PII masking testing |
| operator_phone      | Technician phone number (`+639175551234`) for PII masking testing |

---

# Unified 52-Column Master Schema Strategy

To eliminate schema preview flip-flopping in **Microsoft Fabric Eventstream** and guarantee dropdown field availability in `ManageFields` and SQL Stream Processing nodes, every JSON event payload emitted by the simulator contains the complete set of master schema keys (`UNIFIED_EVENT_SCHEMA_KEYS`).

Attributes not applicable to a specific event category default to JSON `null` (or numeric zero values where appropriate), providing a consistent 52-column payload structure across all 8 stream types.

# Raw Production Telemetry & Data Quality Specification

To simulate an authentic enterprise Industrial IoT streaming environment, raw telemetry payloads produced by the Python Smart Farm Simulator intentionally contain real-world data quality defects and edge transmission anomalies.

The core architectural requirement of this platform is to ingest raw, uncleaned telemetry into **Microsoft Fabric (Eventstream & OneLake Bronze Lakehouse)** and execute all heavy data validation, deduplication, schema normalization, and outlier handling downstream using **PySpark Lakehouse Notebooks (Silver Layer)** and **KQL Database Update Policies**.

## Raw Ingestion Anomaly Categories

| Anomaly Category               | Raw Event Behaviors (Bronze Layer)                                                                                                                             | Downstream Fabric Silver Cleaning Target                                                                             |
| ------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| **Structural Messiness**       | Omitted optional fields, unexpected `null` values, or empty strings (`""`).                                                                                    | Impute default values or filter corrupt records during Silver PySpark ETL.                                           |
| **Formatting Inconsistencies** | Mixed timestamp formats (ISO 8601 UTC strings vs. Unix Epoch integer strings) and inconsistent string casing (`FAC-001` vs. `fac-001`, `ONLINE` vs. `online`). | Standardize timestamps to `TIMESTAMP` type (`to_timestamp()`) and enforce uppercase string casing (`UPPER(TRIM())`). |
| **Out-of-Bounds Data**         | Corrupted sensor readings (`-999.0`), extreme metric spikes (`9999.99`), or invalid range boundaries (e.g. `pH = 14.8`).                                       | Flag out-of-bounds metrics into data quality audit tables or clamp values within physical bounds.                    |
| **Transmission Realities**     | Duplicate event deliveries (~2% batch duplication) and late-arriving events (timestamps shifted 15-120 minutes backwards).                                     | Deduplicate using `dropDuplicates(["event_id"])` or watermark windowing in Spark / KQL.                              |

# Business Entities

Streaming events are emitted by persistent business entities that represent physical assets or operational objects within a smart farming facility.

Each business entity maintains static business metadata, while streaming events capture its operational state at a specific point in time.

| Business Entity      | Event Type           | Description                                                                                       |
| -------------------- | -------------------- | ------------------------------------------------------------------------------------------------- |
| Environmental Sensor | sensor.telemetry     | Produces environmental measurements from growing zones.                                           |
| Equipment Asset      | hardware.metrics     | Produces operational health and performance telemetry.                                            |
| Irrigation Zone      | irrigation.telemetry | Produces irrigation system telemetry, including water delivery and nutrient distribution metrics. |
| Crop Batch           | crop.batch.lifecycle | Produces crop lifecycle events throughout production.                                             |
| Maintenance Activity | maintenance.activity | Records planned and corrective maintenance activities.                                            |
| Platform Service     | platform.system      | Produces simulator and platform operational events.                                               |
| Alert Rule           | alert.critical       | Produces alerts generated from streaming rule evaluation.                                         |

# Event Versioning

All events include the `schema_version` field.

Schema evolution shall follow these principles:

- New optional fields may be added without breaking compatibility.
- Existing fields shall not be renamed.
- Existing fields shall not change data types.
- Breaking schema changes require a new schema version.

# Event Naming Standards

Event types follow the convention:

```

<domain>.<entity>[.<action>]

```

Examples:

- sensor.telemetry
- hardware.metrics
- irrigation.telemetry
- crop.batch.lifecycle

# Architecture Alignment

The Smart Farming Analytics Platform processes every event using the following architecture:

1. Python Smart Farm Simulator publishes events.
2. Eventstream ingests and routes streaming telemetry.
3. Eventhouse stores streaming data for operational analytics.
4. Data Activator evaluates streaming events for alert conditions.
5. Eventhouse continuously persists telemetry into the OneLake Lakehouse Bronze layer.
6. Spark Notebooks transform historical data through the Bronze, Silver, and Gold layers.
7. Fabric Data Factory loads Gold datasets into the Fabric Warehouse.
8. Power BI consumes curated historical datasets for analytical reporting.
