# Functional Requirements

## Purpose

This document defines the functional capabilities required for the Microsoft Fabric Smart Farming Analytics Platform.

The platform shall ingest, process, analyze, and monitor high-volume IoT telemetry generated from multiple smart farming facilities. These requirements serve as the foundation for solution design, implementation, testing, and deployment.

---

# Functional Requirements

## FR-001: Environmental Telemetry Ingestion

### Description

The platform shall continuously ingest environmental telemetry generated by IoT sensors deployed throughout all smart farming facilities.

### Supported Measurements

- Water pH
- Electrical Conductivity (EC)
- Dissolved Oxygen
- Water Temperature
- Ambient Temperature
- Humidity
- Light Intensity

### Acceptance Criteria

- Telemetry is ingested continuously.
- Event timestamps are preserved.
- Source facility metadata is captured.
- Events are available for downstream processing.

---

## FR-002: Equipment Telemetry Ingestion

### Description

The platform shall ingest telemetry generated by facility equipment and mechanical assets.

### Supported Measurements

- Pump Status
- Pump Pressure
- Pump RPM
- Energy Consumption
- Equipment Health Status

### Acceptance Criteria

- Equipment telemetry is streamed continuously.
- Every asset has a unique identifier.
- Equipment history is retained for analytics.

---

## FR-003: Event Routing

### Description

The platform shall route incoming telemetry to appropriate Microsoft Fabric services using Eventstream.

### Destinations

- Eventhouse
- Lakehouse

### Acceptance Criteria

- Valid events are routed successfully.
- Unsupported events are redirected to dead-letter handling.
- Routing failures are logged.

---

## FR-004: Data Quality Validation

### Description

The platform shall validate every incoming telemetry event before downstream storage.

### Validation Rules

- Required fields present
- Valid schema
- Missing values detected
- Duplicate events detected
- Sensor readings within expected biological or operational ranges

### Acceptance Criteria

- Invalid events are flagged.
- Data quality status is assigned.
- Validation metadata is stored.

---

## FR-005: Real-Time Operational Analytics

### Description

The platform shall provide near real-time operational analytics using Eventhouse and KQL Database.

### Operational Views

- Facility Health
- Equipment Health
- Environmental Conditions
- Active Alerts

### Acceptance Criteria

- Live dashboards update automatically.
- KQL queries support operational investigations.
- Operational metrics are available within the target latency.

---

## FR-006: Historical Data Storage

### Description

The platform shall store telemetry in OneLake using a Medallion Architecture.

### Data Layers

- Bronze
- Silver
- Gold

### Acceptance Criteria

- Historical telemetry is retained.
- Data is available for reporting and trend analysis.
- Curated datasets are prepared for downstream analytics.

---

## FR-007: Data Transformation

### Description

The platform shall transform raw telemetry into analytics-ready datasets.

### Transformations

- Cleansing
- Standardization
- Enrichment
- Aggregation
- Data Quality Validation

### Acceptance Criteria

- Silver datasets contain validated telemetry.
- Gold datasets are business-ready.

## FR-008: Dimensional Modeling

### Description

The platform shall organize historical telemetry using a Kimball dimensional model.

### Fact Tables

- fact_sensor_telemetry
- fact_hardware_metrics

### Dimension Tables

- dim_sensor
- dim_crop_batch
- dim_facility_structure
- dim_date
- dim_time

### Acceptance Criteria

- Referential integrity is maintained.
- Star schema supports analytical reporting.

## FR-009: Reporting and Dashboards

### Description

The platform shall provide interactive dashboards for operational and executive users.

### Dashboards

- Executive Dashboard
- Facility Operations Dashboard
- Crop Health Dashboard
- Equipment Health Dashboard

- Interactive filtering is supported.
- Dashboards display current and historical data.

## FR-010: Operational Alerting

### Description

The platform shall automatically generate alerts when predefined environmental or equipment thresholds are exceeded.

### Alert Types

- High Temperature
- Low Dissolved Oxygen
- Abnormal pH
- High Electrical Conductivity
- Pump Failure
- Sensor Offline
- High Energy Consumption

### Acceptance Criteria

- Alerts are generated automatically.
- Alert history is retained.
- Notifications include relevant event context.

---

### Description

The platform shall monitor the health of streaming and batch processing components.

### Monitoring Capabilities

- Pipeline Status
- Streaming Latency
- Failed Events
- Data Quality Metrics
- Processing Throughput

### Acceptance Criteria

- Monitoring metrics are available through dashboards.
- Pipeline failures are logged.
- Processing latency is measurable.

---

## FR-012: Security and Access Control

### Description

The platform shall enforce role-based access control (RBAC) across Microsoft Fabric resources.

- Operations Manager

- Data Analyst

# Assumptions and Constraints

## Assumptions

The following assumptions have been made during solution planning:

- Each IoT sensor generates valid JSON telemetry events.
- All facilities have reliable network connectivity to Microsoft Fabric.
- Microsoft Fabric services are available within the target deployment region.
- Historical telemetry will be retained for long-term analytical reporting.
- Environmental thresholds are predefined by agricultural subject matter experts.
- The Python simulator accurately represents production telemetry patterns.

## Constraints

The initial release of the platform is subject to the following constraints:

- Telemetry data is simulated using Python and Faker rather than physical IoT devices.
- The solution targets Microsoft Fabric as the primary analytics platform.
- Automated equipment control is outside the scope of Version 1.0.
- Predictive machine learning models will not be implemented during the initial release.
- Authentication will rely on Microsoft Fabric workspace security rather than custom identity management.

## Future Enhancements

The following capabilities are intentionally excluded from Version 1.0 but are planned for future iterations:

- Predictive crop yield forecasting
- Equipment failure prediction using machine learning
- Automated irrigation and nutrient control
- Digital twin visualization of farming facilities
- Integration with external weather data providers
- Mobile application for field operators
