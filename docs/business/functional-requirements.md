# Functional Requirements

## Purpose

This document defines the functional capabilities required for the Microsoft Fabric Smart Farming Analytics Platform.

The platform shall ingest, process, analyze, and monitor high-volume IoT telemetry generated from multiple smart farming facilities across the Philippines. These requirements serve as the foundation for solution design, implementation, testing, and deployment.

---

# Functional Requirements

## FR-001: Environmental Telemetry Ingestion

### Description

The platform shall continuously ingest environmental telemetry generated by IoT sensors deployed throughout all smart farming facilities and growing zones.

### Supported Measurements

- Water pH
- Electrical Conductivity (EC)
- Dissolved Oxygen
- Water Temperature
- Ambient Temperature (Air)
- Relative Humidity
- Light Intensity (PPFD / DLI)
- Carbon Dioxide (CO2) Concentration

### Acceptance Criteria

- Telemetry is ingested continuously.
- Event timestamps are preserved.
- Source facility and zone metadata are captured.
- Daytime/Nighttime status (`is_daytime`) and ambient weather indicators are preserved.
- Events are available for downstream real-time and historical processing.

---

## FR-002: Equipment Telemetry Ingestion

### Description

The platform shall ingest telemetry generated by facility equipment and mechanical assets across all facilities.

### Supported Measurements & Attributes

- Equipment ID (`EQ-00001` through `EQ-00512`)
- Equipment Type (`water_pump`, `hvac`, `dosing_system`, `led_lighting`, `chiller`, etc.)
- Operating Status (`ONLINE`, `DEGRADED`, `OFFLINE`, `WARNING`, `ERROR`)
- Equipment Health Score (`0.0` to `100.0`)
- Current Load Percentage (`0.0%` to `100.0%`)
- Failure Probability (`0.0` to `1.0`)
- Cumulative Runtime Hours
- Vibration (`vps`)
- Operating Temperature (`°C`)
- Power Consumption (`kW`)

### Acceptance Criteria

- Equipment telemetry is streamed continuously.
- Every asset has a unique identifier linked to its assigned facility and zone.
- Equipment wear and health trends are retained for analytical reporting.

## FR-003: Crop Lifecycle & Crop Telemetry Tracking

### Description

The platform shall track biological crop batch growth profiles, growth stage transitions, and crop condition telemetry.

### Stage Transitions & Metrics

- Lifecycle Stages: `GERMINATION`, `SEEDLING`, `VEGETATIVE`, `MATURE`, `HARVESTED`
- Metrics: `age_days`, `health_score`, `growth_rate`, `biomass_grams`, `water_consumption_liters`, `nutrient_consumption_grams`, `environmental_stress_index`

### Acceptance Criteria

- Crop stage transitions emit lifecycle events.
- Biomass accumulation and stress index are tracked continuously.

---

## FR-004: Maintenance Work Order Lifecycle Management

### Description

The platform shall track routine and preventative maintenance work order execution across equipment assets and growing zones.

### Work Order Attributes

- Work Order ID (`WO-00001` through `WO-00010`)
- Maintenance Type (`PREVENTATIVE`, `CORRECTIVE`, `CALIBRATION`, `EMERGENCY`)
- Maintenance Status (`IN_PROGRESS`, `COMPLETED`)
- Assigned Technician (`tech.zone-001@smartfarm.ph`)
- Duration Tracking (`estimated_duration_minutes`, `remaining_duration_minutes`, `completion_percent`)
- Health Restoration Value & Technician Notes

### Acceptance Criteria

- Maintenance events are emitted on state progression.
- Technician notes and duration metrics are captured for operational reporting.

---

## FR-005: Regional Philippine Facility Metadata Integration

### Description

The platform shall integrate rich regional facility dimension metadata representing 8 vertical farming facilities across the Philippines.

### Supported Facility Dimension Metadata

- `facility_id` (`FAC-001` through `FAC-008`)
- `facility_name`
- `region` (NCR, Calabarzon, Cordillera, Davao Region, Central Visayas, Northern Mindanao, Soccsksargen)
- `city` (Manila, Calamba, La Trinidad, Davao City, Cebu City, Cagayan de Oro, General Santos)
- Geographic Coordinates (`latitude`, `longitude`, `elevation_m`)
- Operational Metadata (`water_source`, `power_grid_redundancy`, `max_zone_capacity`)

### Acceptance Criteria

- Facility dimension data is ingested and integrated into `DimFacility` in the Silver and Gold analytics layers.
- Supports geospatial map visualizers and regional performance slicing in Power BI.

## FR-006: Raw Data Anomaly Ingestion & PII Governance

### Description

The platform shall ingest raw data quality defects and mock operator PII fields to validate Microsoft Fabric Medallion data cleansing and Purview governance rules.

### Defect & PII Types

- PII Attributes: `operator_contact` (`tech.fac-001@smartfarm.ph`), `operator_phone`
- Defects: Duplicate payload bursts, missing values (`null`, `"N/A"`), format standardization variations, stringified numbers, thermal/pH outliers, and orphaned foreign keys (`EQ-99999_ORPHAN`).

### Acceptance Criteria

- Raw dirty telemetry lands untouched in the Bronze Lakehouse.
- PySpark Silver cleaning jobs execute SHA-256 PII hashing (`sha2(operator_contact, 256)`), deduplication (`dropDuplicates(["event_id"])`), and outlier quarantine filtering.

---

## FR-007: Event Routing & Unified Master Schema Padding

### Description

The platform shall route incoming telemetry to Microsoft Fabric using Eventstream. To guarantee schema stability, every event payload emitted by the simulator contains all 52 master schema keys (`UNIFIED_EVENT_SCHEMA_KEYS`).

### Routing Destinations

- Fabric Eventhouse (KQL Database `SmartFarmingKQLDB`) via SQL Stream Processing Nodes.
- Fabric Medallion Lakehouse (Bronze Delta Tables).

### Acceptance Criteria

- Valid events are routed successfully to Eventhouse tables (`EnvironmentalTelemetry`, `EquipmentTelemetry`, etc.).
- Eventstream SQL nodes execute type casting (`CAST(val AS FLOAT)`) and stream filtering (`WHERE event_type = ...`).

---

## FR-008: Real-Time Operational Analytics

### Description

The platform shall provide near real-time operational analytics by querying streaming telemetry stored in Eventhouse using KQL.

The platform shall expose this operational data through a Power BI Real-Time Dashboard for monitoring live farming operations.

### Operational Views

- Live Sensor Telemetry & Day/Night Dynamics
- Equipment Wear & Failure Probability
- Environmental Threshold Breaches
- Active Maintenance Work Orders & Technician Velocity
- Facility Regional Operational Health

---

## FR-009: Historical Data Storage & Medallion Architecture

### Description

The platform shall store telemetry in OneLake using a Medallion Architecture (Bronze, Silver, Gold).

---

## FR-010: Dimensional Modeling & Warehouse Direct Lake

### Description

The platform shall organize historical telemetry using a Kimball dimensional star schema (`FactTelemetry`, `FactCropGrowth`, `FactEquipmentMaintenance`, `DimFacility`, `DimCrop`, `DimEquipment`, `DimDate`).

Exposed through Synapse Data Warehouse via **Direct Lake** mode for zero-copy Power BI performance.

---

## FR-011: Operational Alerting & Data Activator

### Description

The platform shall automatically generate real-time alerts via Data Activator (Reflex) when environmental or equipment failure thresholds are breached (`failure_probability > 0.35`, `water_ph < 5.5`).

# Assumptions and Constraints

## Assumptions

The following assumptions have been made during solution planning:

- Each IoT sensor generates valid JSON telemetry events.
- All facilities have reliable network connectivity to Microsoft Fabric.
- Microsoft Fabric services are available within the target deployment region.
- Historical telemetry will be retained for long-term analytical reporting.
- Environmental thresholds are predefined by agricultural subject matter experts.
- The Python simulator generates realistic telemetry patterns representative of production IoT devices.

## Constraints

The initial release of the platform is subject to the following constraints:

- Telemetry is generated by a configurable Python simulator using Faker rather than physical IoT devices.
- The solution targets Microsoft Fabric as the primary analytics platform.
- Automated equipment control is outside the scope of Version 1.0.
- Predictive machine learning models will not be implemented during the initial release.
- Authentication will rely on Microsoft Fabric workspace security rather than custom identity management.
